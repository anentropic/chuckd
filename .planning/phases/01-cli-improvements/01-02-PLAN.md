After Plan 01, ChuckD.java has these changes:
```
Usage: chuckd [-hqV] [-c=<compatibilityLevel>] [-f=<schemaFormat>]
              [-l=<logLevel>] [-o=<outputFormat>] <schemaArgs>...

Positional args (schemaArgs):
  Glob mode (1 arg): pass a quoted glob pattern
  Explicit mode (2+ args): <previous...> <new> -- last arg is the new schema

New options:
  -q, --quiet    Suppress file metadata output on stderr
  -o, --output   TEXT (default) or JSON

Exit codes:
  0   Compatible (or trivially compatible with a single glob match)
  1   Incompatible -- breaking changes detected
  2   Usage error -- bad arguments, missing files, or glob matches nothing
  3   Runtime error -- file I/O failure or schema parse error
```

    **2. Add new typed exit code tests:**

    ```bash
    @test "bad flag -> exit 2" {
      run "${bin_path}/chuckd" --nonexistent-flag
      [ "$status" -eq 2 ]
    }

    @test "missing file -> exit 3" {
      run "${bin_path}/chuckd" "nonexistent-file-1.json" "nonexistent-file-2.json"
      [ "$status" -eq 3 ]
    }
    ```

    **3. Add glob mode tests:**

    ```bash
    @test "glob mode, no matches -> exit 2" {
      run "${bin_path}/chuckd" "nonexistent-dir/*.json"
      [ "$status" -eq 2 ]
      [[ "$output" = *"No files matched"* ]] || [[ "${lines[0]}" = *"No files matched"* ]]
    }

    @test "glob mode, single match -> exit 0" {
      # Create a temp dir with a single file to glob
      tmpdir=$(mktemp -d)
      cp "${res_path}/jsonschema/person-base.json" "${tmpdir}/schema-v1.json"
      run "${bin_path}/chuckd" "${tmpdir}/schema-v1.json"
      rm -rf "${tmpdir}"
      [ "$status" -eq 0 ]
    }
    ```

    Note: A full glob mode smoke test with 2+ matches requires multiple schema files in a directory that match a glob and produce a meaningful compatibility result. Create a temp dir, copy test schemas with sortable names (e.g. `person-v1.json`, `person-v2.json`), run with glob pattern.

    **4. Add --quiet flag test:**

    ```bash
    @test "quiet mode suppresses stderr metadata" {
      run "${bin_path}/chuckd" -q -c BACKWARD "${res_path}/jsonschema/person-base.json" "${res_path}/jsonschema/person-narrowed.json"
      [ "$status" -eq 0 ]
      # BATS captures both stdout and stderr in $output by default
      [ "$output" = "" ]
    }
    ```

    **5. Add JSON output test:**

    ```bash
    @test "JSON output on compatible schemas produces []" {
      run "${bin_path}/chuckd" -q -o JSON -c BACKWARD "${res_path}/jsonschema/person-base.json" "${res_path}/jsonschema/person-narrowed.json"
      [ "$status" -eq 0 ]
      [[ "$output" = *"[]"* ]]
    }
    ```

    IMPORTANT: BATS `run` captures both stdout and stderr into `$output`. For tests that check only stdout or only stderr, use appropriate BATS techniques or accept that metadata appears in $output. For the `--quiet` test this is fine since quiet suppresses stderr and compatible schemas produce no stdout in TEXT mode. For the JSON test, use `--quiet` flag to suppress stderr metadata so only the JSON `[]` appears in output.
